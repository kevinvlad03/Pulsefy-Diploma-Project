## 0. Quick orientation

- **Faculty / format:** UPB FILS, Bachelor's diploma project. Two title pages in English + one in Romanian + a signed Diploma Project Theme form + signed Academic Honesty Statement + TOC + Table of Figures + 6 chapters + Conclusions + References + Glossary.
- **Page target:** ~80 pages of body content. IARCA has 81. Aim 75–90.
- **Figure target:** 60–75 figures. IARCA has 75. The thesis is **figure-heavy** — diagrams are the primary depth signal, not prose.
- **Voice:** third-person impersonal in body chapters; first-person creeps in inside the Conclusions chapter. Plain ESL-grade English clarity is acceptable; over-polished GPT prose is a tell. Sentence-case for headings, title-case for figure/table captions.
- **Citation style:** numbered footnotes at the page bottom for inline claims (Ch 1–2). Numbered `[N]` reference list at the end. Both styles coexist — see §11.

## 2. Body shape — chapter-by-chapter

### Chapter 1 — Introduction (~4–6 pages, 4 subsections)

Subsection structure (sentence case):

- **1.1 Music everywhere** *(IARCA: "Documents everywhere")*
- **1.2 Motives for using an AI music platform** *(IARCA: "Motives for using a Document Management System")*
- **1.3 Advantages of AI-assisted music discovery and creation** *(IARCA: "Advantages of automated data extraction")*
- **1.4 Structure of the paper** *(unchanged title)*

**Per-subsection target:**
| Sub | Paragraphs | ~Words | Figure | Bullets |
|---|---|---|---|---|
| 1.1 | 4–5 | 300–340 | 1 (Fig 1: streaming/creator-economy stat or "manual playlist vs algorithmic discovery") | none |
| 1.2 | 4–5 | 230–260 | 0 | none |
| 1.3 | 7–9 | 350–400 | 1 (Fig 2: "manual curation/production vs AI-assisted") | none |
| 1.4 | 3–4 + 1 list | 140–160 | 0 | one bulleted list of the 7 chapters |

**Opening cadence (study before drafting):** IARCA opens 1.1 with `In these days, as the world is modernizing, digitalization gets more and more attention.` — broad cultural fact + qualified amplification. Use this rhythm for the music-everywhere opener.

**Closing cadence:** every section ends with a sentence that *reaches forward* toward the next subsection's topic. No standalone wrap-ups.

**Citations:** any statistic in 1.1–1.3 needs a numbered footnote at the bottom of its page (industry reports, Spotify revenue numbers, MusicGen paper, etc.). One stat = one footnote.

**1.4 specific copy pattern:** name the project, list the 7 chapters in a bullet list, end with attributions of design tools used (`The diagrams in this paper were created using draw.io (https://www.diagrams.net/), Visual Paradigm (...), and Canva (https://canva.com/)`).

### Chapter 2 — State of the art (~12 pages, 3 subsections)

- **2.1 Scientific state of the art**
  - **2.1.1** *Music Information Retrieval (audio embeddings / classification / retrieval)*
  - **2.1.2** *Generative audio models (MusicGen, AudioLDM, transformer-based audio generation)*
- **2.2 Commercial state of the art** — pick exactly **four** tools to mirror IARCA's four. Suggested mix:
  - **2.2.1** Spotify (or Apple Music / YouTube Music) — incumbent recommendation engine
  - **2.2.2** Suno (or Udio) — user-facing generative tool
  - **2.2.3** ElevenLabs (or Mubert API) — developer/API platform with audio generation
  - **2.2.4** Splice (or BandLab SongStarter / Soundraw) — creator-side / DAW-adjacent
- **2.3 Summary and comparison of existing approaches** — the comparison table goes here

**2.2.x competitor template (mandatory shape — same for all four):**
1. **Identity sentence:** `<Tool name> is a <type of solution> [provided/offered] by <company>, launched in <year>.`
2. **Workflow / how it works:** 1–2 paragraphs. One figure showing the workflow (UI screenshot or vendor diagram). No inline "as shown in Figure X" — let the figure follow the prose.
3. **Quantitative outcomes:** percentage savings, throughput, MAU, generation speed, user count — whatever public numbers the vendor publishes. Footnote each stat.
4. **Integration / pricing / API notes:** API availability, EUR/USD per call or per month, ecosystem (e.g. Spotify-for-Developers, Hugging Face for MusicGen).

**2.3 comparison table (Table 2 of the thesis — Table 1 is the receipt-fields-style table, optionally borrowed for §2.2.x).** 4 rows × ~6 columns. Suggested criteria columns: `Pricing` / `API access` / `Self-host` / `Open source` / `Generative output formats` / `Latency` / `Customisation depth`. Caption: `Table N: Comparison of commercial AI music platforms`.

**Voice in this chapter:** factual, no marketing adjectives. IARCA never writes "vibrant" or "breathtaking" about competitors — neither should we.

**Footnote count target:** 6–10 numbered footnotes in this chapter.

### Chapter 3 — Requirements analysis (~17 pages, 6+1 subsections)

This is the most templated chapter. Clone IARCA's structure exactly.

- **3.1 Functional requirements**
  - **3.1.1 Actors and agents** — 2 short prose paragraphs. For Pulsefy use **3 actors**: Visitor (anonymous), Authenticated User, Administrator. Free vs Premium tier handled as a precondition on premium-only use cases (`The user must have a premium subscription tier`), not as a fourth actor — keeps the diagram clean.
  - **3.1.2 Software use case diagram** — Figure 9 (one diagram, no prose). Two-actor or three-actor stick figures + System rectangle + 11–14 use case ovals + `<<includes>>` / `<<extends>>` lowercase labels. Tool: **draw.io** (default UML use case template).
  - **3.1.3 Use case functional requirements** — pure list, no prose. One sub-sub-section per UC: `3.1.3.X <Verb-phrase title> – UCx`. Under each, a lettered list of "System does X – N" lines. **Decision needed:** the trailing integer (`– 10`, `– 6`, `– 4`) is unexplained in IARCA's text and looks like a MoSCoW priority. Either **(a)** add a one-paragraph note in 3.1.3 explaining the scale (e.g. "values reflect MoSCoW priority: 10 = must, 6 = should, 4 = could, 1 = won't"), or **(b)** drop the integers entirely. Don't replicate the unexplained number.
  - **3.1.4 Use case descriptions and system sequence diagrams** — 1 description table + 1 SSD per UC. The **canonical 7-field UC table** to clone:

    ```
    | Use Case Name:         | <Imperative verb phrase>            |
    | Scope:                 | System                              |
    | Actor:                 | <Role>                              |
    | Preconditions:         | <Single sentence, present tense.>   |
    | Post-conditions:       | <Single sentence, perfective.>      |
    | Main success scenario: | (nested 2-col table)                |
    |                        | | <Actor>      | System          |   |
    |                        | | 1. ...       |                 |   |
    |                        | |              | 2. ...          |   |
    |                        | | 3. ...       |                 |   |
    |                        | |              | 4. ... – A1     |   |
    | Extension:             | A1. <Trigger>.                      |
    |                        | <Recovery instruction sentence.>    |

    Table N: <Use Case Name> Description
    Figure N+1: <Use Case Name> System Sequence Diagram
    ```

    Rules:
    - Step text is **lowercase imperative** (`access website`, `show login form`). No periods inside step cells.
    - Branch markers in steps trail with `– A1`, `– A2` referring to Extension flows.
    - Postconditions read as state assertions (`The user is logged in and the home page is displayed.`).
    - When a UC has no alt flow, **omit the Extension row entirely**.
    - **Zero connecting prose** between UCs. Heading → table → diagram → next heading.
  - **3.1.5 Activity diagrams** — only for the 2–3 most branching use cases. Pulsefy candidates: **Generate AI music** (model selection → tier check → queue → success/failure), **Upload video and overlay audio** (file validation → encoding → overlay → preview), optionally **Register** (email send → verify → activate). Swimlane format with two columns (Actor | System), draw.io default colours (yellow rounded actions / yellow diamond decisions / red arrows / black start/end nodes).
  - **3.1.6 Operation contracts** — 3–4 contracts for the most complex backend operations (the ones that trigger Python subprocesses or external API calls). Format = three borderless tables, fields in this exact order: `Operation Name:` (code-style signature with parameters), `Use Case Name:` (back-reference like `Generate AI music – UC6`), `Preconditions:` (single sentence), `Post-conditions:` (multi-line stack of state-change sentences, no bullets). Pulsefy candidates: `GenerateTrack(modelId, prompt, duration, userId)`, `OverlayAudioOnVideo(videoId, trackId, offset)`, `FetchRecommendations(userId, seedTrackId)`, `ChargeSubscription(userId, plan)`.
- **3.2 Non-functional requirements** — 5–8 sub-paragraphs, each one category, 2–4 sentences. Recommended set for Pulsefy: **Performance**, **Security**, **Usability**, **Reliability**, **Scalability**, **Maintainability**, **Portability**. Each gets a short defended claim (e.g. *Performance — track playback should start within 1.5 s of click; AI generation should return within 30 s for free tier and within 90 s for premium MusicGen calls.*).

|---|---|---|---|
| UC1 | Register | Visitor | optional |

### Chapter 4 — Design (~16 pages, 2 subsections)

- **4.1 Architecture**
  - **4.1.1 Architecture choice** — for Pulsefy this is **client-server with separate Python ML services**, not strict monolithic. Adapt IARCA's defensive opening pattern: *definition → drawback → "but this fits because… + here's the escape hatch"*. The escape hatch here is the Python subprocess (MusicGen, gTTS) and the Pollinations external API. Three short paragraphs (~180 words) plus an architecture block diagram (Figure analog of 23) showing: Browser → React/Vite frontend → Express/Node API → Postgres → arrows out to: MusicGen Python subprocess, gTTS Python subprocess, Pollinations.ai HTTPS, Jamendo HTTPS.
  - **4.1.2 Package diagrams** — zero prose between diagrams. Two figures stacked:
    - Figure A: `pulsefy.server` packages — `routes/`, `services/`, `middleware/`, `db/`, `utils/`, plus the Python `musicgen/` and `tts/` directories shown as separate package nodes.
    - Figure B: `pulsefy.client` packages — `pages/`, `components/` (with sub-package `ui/` for shadcn), `hooks/`, `lib/`. Plus a dashed dependency arrow from client → server.
  - **4.1.3 Deployment** — describe local dev deployment (Vite on :8080, Node on :4000, Postgres locally) and any planned cloud target (DigitalOcean droplet? Railway? "Local only for thesis demo, production deployment out of scope"). One screenshot of the build command output or `package.json` scripts. IARCA used IIS publish dialogs — Pulsefy can use a screenshot of `npm run build` output or a `Dockerfile` if one exists.
- **4.2 Detailed design**
  - **4.2.1 Design class diagrams and design sequence diagrams** — one DCD + one DSD per use case (~11 pairs = ~22 figures). Each subsection title is just `4.2.1.X UCx` (no descriptive name — IARCA convention). Inside each: DCD first, DSD second, **zero prose**. Captions read `Figure N: <UC name> Design Class Diagram` / `Figure N+1: <UC name> Design Sequence Diagram`.

    For DCDs, show: View component → Service class (Angular/React) → Controller (Express route) → Repository / DB layer → Model class. Methods listed with TS-style signatures (`isLoggedIn(): boolean`, `login(email: string, password: string): Promise<AuthResponse>`).

    For DSDs, show vertical lifelines for the same classes, multi-level numbered messages (`1.5.1.5: GenerateToken(userId: Guid = authResponse.UserId): string`), and `alt` blocks for branching.

    UCs with shared state can collapse — IARCA merged Add/Modify into one DCD with two DSDs (UC10). For Pulsefy, **Generate AI music free vs premium** is a candidate to collapse: one DCD, two DSDs (one branching to Pulsefy AI subprocess, one branching to MusicGen).
  - **4.2.2 Database structure** — **mirror IARCA's three-ERDs trick.** Open with: *"The structure of the Pulsefy database can be analysed from three different points of view."*
    - **ERD 1 — Plug-in / AI subsystem schema** (the tables only the generation pipeline touches). Pulsefy candidates: `ai_generations`, `ai_models` (if you add one for tier metadata), `tts_generations` (if persisted).
    - **ERD 2 — Web app at deployment** (8 tables, FKs but no rows): `users`, `tracks`, `playlists`, `playlist_tracks` (junction), `listening_events`, `recommendations`, `ai_generations`, `follows` (junction). **Two M:N junction tables** — explicitly mention this in prose ("Pulsefy has two many-to-many relations: a user has many playlists which contain many tracks, and users follow other users").
    - **ERD 3 — Web app at runtime** (same schema + example rows showing the FK joins for one user's listening history → recommendations → AI generations). This serves as a "populated example" rather than a dynamic-table-creation trick (IARCA had per-model tables created at runtime — Pulsefy doesn't).
    - One `Table N: Default Columns / Schema details` if useful (optional).

### Chapter 6 — Validation (~1 page)

This is *short* (one page in IARCA). The trick: report a single concrete validation source plus one infographic figure. Pulsefy doesn't have a real-world deployment, so the validation strategy must be honest and credible.

**Recommended Pulsefy validation strategy (combine three sources for thesis-grade depth):**

1. **Generation latency benchmarks (primary)** — run N=200 prompt requests through both Pulsefy AI free and MusicGen premium. Report p50, p95, p99 wall-clock time and CPU usage. Concrete numbers carry weight.
2. **Small listening study (secondary)** — recruit 10–20 classmates, blind A/B "which track is AI generated, which sounds more musical", report agreement scores and inter-rater reliability. Methodologically modest, but parallel to IARCA's anecdotal "client feedback was satisfying".
3. **System capacity metrics (supporting)** — concurrent users supported (load test with `k6` or `autocannon`), DB query p95 latency, video upload throughput. Mention any breaking points.

**Page shape:** opening sentence ~12 words (`In this chapter we focus on the validation of the Pulsefy platform.`), 2–3 paragraphs of methodology + numbers, **one infographic Figure 75-analog** summarising five headline numbers (e.g. *200 prompts generated*, *94% completed without retry*, *avg 47 s p95 for MusicGen*, *15 listening-study participants*, *82% correct guess on AI-vs-human*). Caption format `Figure N: Pulsefy validation summary` (italic-bold, matching IARCA's Fig 75 styling).

**Hedging language (good):** *"In absence of production deployment, the system was validated through latency benchmarks and a small listening study with N=15 participants."*

**Avoid:** "state of the art", "industry-leading", "proves", "demonstrates without doubt".

### Conclusions (~1 page, no chapter number)

Mirror IARCA's exact 5-block structure:

1. **Recap thesis claim** — Pulsefy provides AI music discovery + creation in one product, addressing a gap left by single-purpose tools (Spotify-style discovery vs Suno-style generation).
2. **Bullet list of design constraints** surfaced during analysis (≤6 bullets): simplicity for first-time users, sub-second playback start, dual-model fairness across tiers, copyright-safe generation, cross-browser compatibility, mobile-responsive UI.
3. **One personal sentence in first person:** *"I implemented Pulsefy with these constraints in mind…"* (IARCA does this transition explicitly — keep it short, do not overdo first-person elsewhere).
4. **Limitations + future work** — single-language UI, MusicGen GPU costs preventing production launch, recommendation system uses simple rule-based heuristics rather than embeddings, no native mobile app, no production deployment yet.
5. **Generic outro** — keep this **short**, one or two sentences. Resist the urge to write a TED-talk paragraph. IARCA's outro is a tell ("there is a need in this modernizing world for automatization") — bake in a quieter version: *"The intersection of streaming and generation is moving fast, and Pulsefy aims to be a small, honest contribution to it."*

**Voice:** modest-confident with first-person allowed. Acknowledge the rough start ("constant support was needed", IARCA-style honesty) without sounding defeated.

## 3. Voice & style rules (humanizer-aware)

Read these at the start of every writing session.

### Prose
- **Third person impersonal in body chapters.** Never "we" except inside the Conclusions chapter (one sentence) and never "you". Allow occasional implicit `the user`, `the system`, `the application`.
- **Sentence length: short and medium, varied.** Avoid mega-sentences. Avoid uniform-length staccato.
- **Plain "is" copula.** Drop "stands as / serves as / represents". *Pulsefy is a music platform.* ✓ *Pulsefy serves as a music platform.* ✗
- **No promotional adjectives.** Ban these: vibrant, breathtaking, seamless, robust, comprehensive, cutting-edge, groundbreaking, innovative (as filler), pivotal, intricate, profound, transformative.
- **No AI vocabulary.** Ban these: delve, leverage (verb), navigate (metaphorical), foster, underscore, highlight (verb), tapestry, landscape (abstract), ecosystem (abstract), align with, in the realm of, in today's [adj] world.
- **No filler / signposting.** Cut: "It is important to note that…", "It is worth mentioning…", "Let's explore / dive into / break down…", "First and foremost…".
- **No rule-of-three rhetorical pattern.** Functional 3- or 4-item lists are fine ("React, Vite, and Tailwind"); rhetorical "X, Y, and ultimately Z" is not.
- **No em dashes.** Use commas, periods, parentheses. (En-dashes inside captions like `Pulsefy – Login Popup` are fine — they're separators, not prose.)
- **No bold or italics inside paragraph prose.** Bold is reserved for table-cell labels and section headers (Word's heading styles). Italics for foreign-language titles, paper titles, and figure-caption emphasis (Fig 75 style).
- **Sentence-case for headings.** *Use case descriptions and system sequence diagrams* ✓ *Use Case Descriptions And System Sequence Diagrams* ✗
- **Title-case for figure and table captions.** *Figure 12: Upload Documents System Sequence Diagram* ✓
- **Straight quotes.** No curly typographic quotes in body text. (IARCA uses curly — Pulsefy can either match IARCA or use straight; per the humanizer skill rule 19, **prefer straight quotes**.)

### Citations
- Every numerical claim outside the user's own measurements gets a footnote in Ch 1–2.
- Vendor pricing must come from the vendor site (footnote with retrieval date).
- ML claims come from the source paper (audiocraft, MusicGen arXiv, Lerch MIR textbook).
- No "experts argue / studies show" without a real citation.

### Diagrams
- **Tool:** draw.io (default styles, default colours) for UML use case, sequence, activity, class diagrams. Visual Paradigm optional. **Avoid** PlantUML output — the auto-layout has GPT-era visual tells.
- **No screenshots with red arrows / numbered callouts.** Prose directs attention.
- **Captions below figures, centred, plain text** (`Figure N: Caption`). Italic-bold reserved for the headline validation figure (Ch 6).
- **No internal "Code listing N" labels** — pseudocode blocks sit inline without numbered labels.

### Pseudocode style
- Dark background, syntax-highlighted, monospace.
- Keywords UPPERCASE: `FUNCTION`, `IF`, `THEN`, `ELSE IF`, `ENDIF`, `WHILE`, `FOR`, `CALL`, `RETURN`, `END FUNCTION`.
- Variables camelCase: `pdfDoc`, `firstPage`, `operationId`.
- `// inline comments` in green.
- 4-space indent.
- ~25 lines max per block.
- One block per phase in Ch 5.2.

### Final humanizer pass (mandatory before submission)
Run *every chapter's final draft* through the audit prompt from `/Users/dumitruvlad/.claude/skills/humanizer/SKILL.md`:
